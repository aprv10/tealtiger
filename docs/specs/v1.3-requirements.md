# Requirements Document — TealTiger v1.3.0 "Governance Bundle"

> **STATUS: DRAFT** — This document is under active iteration. Requirements may change based on review feedback.

## Introduction

TealTiger v1.3.0 is a major capability expansion that evolves the SDK from v1.2.0's runtime enforcement primitives into an enterprise-ready governance layer for autonomous AI agents. The release bundles 10 epics addressing: autonomous governance modes, non-human identity governance, secure change governance, high-impact safety rails, SOC/IR evidence pipelines, standards-aligned policy packs, cryptographic governance receipts, declarative governance workflows, OWASP LLM 2026 gap closure, and lightweight ML detection.

**Design Principles:**
- No LLM in the governance path (deterministic decisions only)
- A classifier is NOT an LLM (optional, local, single-purpose, deterministic)
- SDK-only architecture (no infrastructure dependency)
- TypeScript + Python parity
- Apache 2.0 open source
- Backward compatible with v1.2.0

**Benchmark Targets:**
- Garak jailbreak detection: 85%+ (from 40%)
- PINT F1: 80%+ (from 54.5%)
- Precision: maintain 85%+

## Glossary

- **TealEngine**: The deterministic policy decision point (PDP) that evaluates governance policies and produces typed Decision objects
- **TealGuard**: The input/output guardrail module providing content filtering, PII detection, and prompt injection detection
- **TealMemory**: The memory governance module managing 5 scopes and 4 classifications for agent memory operations
- **TealRegistry**: The model/tool registry module providing allowlisting, provenance verification, and supply chain governance
- **TealAudit**: The audit logging module producing versioned, PII-redacted governance logs
- **TealSecrets**: The secret detection module identifying credentials and sensitive data in content
- **TealCircuit**: The circuit breaker module preventing cascading failures
- **TealProof**: The cryptographic governance receipt module producing Merkle-tree-based tamper-evident proof chains
- **TealFlow**: The declarative governance workflow engine using YAML-based event-driven pipelines
- **TealClassifier**: The optional lightweight ML classifier module for prompt injection detection using ONNX inference
- **TealDrift**: The behavioral drift detection module comparing agent behavior against statistical baselines
- **Policy_Decision_Point (PDP)**: The component that evaluates a governance request against loaded policies and returns a deterministic Decision
- **Decision**: A typed object containing action (allow/deny/modify), risk score, reason codes, evidence fields, and policy version
- **NHI (Non-Human Identity)**: An AI agent or automated system treated as a principal with identity, ownership, scope, and revocation lifecycle
- **PLAN_ONLY Mode**: A runtime mode where the agent may reason and plan but the governance layer blocks all side-effecting actions
- **FREEZE**: An immutable governance control that blocks specified actions regardless of agent output or policy evaluation
- **Automation_Level**: A policy metadata field specifying the degree of autonomy for an action category (auto-deny, auto-sanitize, auto-allow, approval-required)
- **Governance_Receipt**: A cryptographic hash of a policy decision combined with context, timestamp, and previous hash, forming a Merkle tree leaf
- **Merkle_Root_Anchor**: A periodic commitment of the Merkle tree root to an RFC 3161 timestamping service or transparency log
- **TealFlow_Workflow**: A YAML document defining event triggers, jobs, steps, conditionals, and dependencies for governance pipelines
- **Ensemble_Mode**: A configuration determining how regex and ML classifier signals are combined (regex_only, ml_only, ensemble_union, ensemble_intersection)
- **ONNX_Runtime**: The cross-platform inference engine used to run the TealClassifier model locally without Python dependency
- **MCP (Model Context Protocol)**: A protocol for tool definitions consumed by AI agents; subject to definition-drift attacks
- **SIEM**: Security Information and Event Management system receiving governance decision logs
- **OTel**: OpenTelemetry — the observability framework used for span conventions and trace export
- **OWASP_Agentic_Top_10**: The OWASP Top 10 for Agentic Applications standard defining common agentic security risks
- **EARS**: Easy Approach to Requirements Syntax — a pattern-based method for writing unambiguous requirements
- **TEEC**: Typed Evidence & Evidence Contract — the structured evidence format produced by TealEngine v1.2

### Requirement 1: Autonomous Governance Modes (Epic A)

**User Story:** As a platform engineer, I want to assign automation levels to action categories in governance policies, so that autonomous agent workflows can proceed at machine speed for low-risk actions while requiring approval for high-risk actions.

#### Acceptance Criteria

1. THE Policy_Decision_Point SHALL support an `automation_level` metadata field on each policy rule with values: `auto_deny`, `auto_sanitize`, `auto_allow`, and `approval_required`.
2. WHEN a governance request matches a rule with `automation_level` set to `auto_allow`, THE Policy_Decision_Point SHALL return an allow Decision without requiring external approval.
3. WHEN a governance request matches a rule with `automation_level` set to `approval_required`, THE Policy_Decision_Point SHALL return a pending Decision that blocks execution until an external approval signal is received.
4. WHEN a governance request matches a rule with `automation_level` set to `auto_deny`, THE Policy_Decision_Point SHALL return a deny Decision immediately.
5. WHEN a governance request matches a rule with `automation_level` set to `auto_sanitize`, THE Policy_Decision_Point SHALL return a modify Decision with sanitized content.
6. THE Policy_Decision_Point SHALL support hot-swapping policy bundles at runtime without restarting the host process.
7. WHEN a policy bundle hot-swap fails validation, THE Policy_Decision_Point SHALL retain the previously active bundle and emit an error event.
8. THE Policy_Decision_Point SHALL produce identical Decision outputs for identical inputs regardless of automation_level (deterministic evaluation).

### Requirement 2: Non-Human Identity Governance (Epic B)

**User Story:** As a security architect, I want to govern AI agents as principals with identity, ownership, and scoped capabilities, so that agent permissions are auditable and revocable like human identities.

#### Acceptance Criteria

1. THE TealEngine SHALL accept an NHI identity descriptor containing: agent identifier, owner, creation timestamp, capability scope, and environment constraints.
2. WHEN an NHI identity descriptor is provided in a governance request, THE Policy_Decision_Point SHALL evaluate policies against the NHI's declared capability scope.
3. WHEN an NHI attempts an action outside its declared capability scope, THE Policy_Decision_Point SHALL deny the action with reason code `NHI_SCOPE_VIOLATION`.
4. THE TealEngine SHALL support an NHI inventory schema that records all registered agent identities, their owners, and their current status (active, suspended, revoked).
5. WHEN an NHI identity status is set to `revoked`, THE Policy_Decision_Point SHALL deny all actions from that identity with reason code `NHI_REVOKED`.
6. WHEN an NHI identity status is set to `suspended`, THE Policy_Decision_Point SHALL deny all actions from that identity with reason code `NHI_SUSPENDED`.
7. THE TealEngine SHALL support environment constraints on NHI identities that restrict which environments (production, staging, development) the agent may operate in.
8. WHEN an NHI attempts an action in an environment not listed in its environment constraints, THE Policy_Decision_Point SHALL deny the action with reason code `NHI_ENVIRONMENT_VIOLATION`.

### Requirement 3: Secure Change and Patch Governance (Epic C)

**User Story:** As a development team lead, I want governance controls over AI code agents that propose patches and pull requests, so that automated code changes are constrained to approved directories, branches, and approval workflows.

#### Acceptance Criteria

1. THE Policy_Decision_Point SHALL recognize an action class `CODE_CHANGE` with attributes: target file paths, target branch, change type (create, modify, delete), and diff hash.
2. WHEN a CODE_CHANGE action targets a file path not in the configured allowlist, THE Policy_Decision_Point SHALL deny the action with reason code `CODE_CHANGE_PATH_VIOLATION`.
3. WHEN a CODE_CHANGE action targets a branch not in the configured branch allowlist, THE Policy_Decision_Point SHALL deny the action with reason code `CODE_CHANGE_BRANCH_VIOLATION`.
4. WHERE the two-person rule option is enabled, THE Policy_Decision_Point SHALL require approval from a second identity before allowing a CODE_CHANGE action.
5. THE Policy_Decision_Point SHALL produce an evidence artifact for each CODE_CHANGE decision containing: diff hash, policy version, decision outcome, and approval identities (if applicable).
6. IF a CODE_CHANGE action fails to include a diff hash, THEN THE Policy_Decision_Point SHALL deny the action with reason code `CODE_CHANGE_MISSING_EVIDENCE`.

### Requirement 4: High-Impact Safety Rails (Epic D)

**User Story:** As a CISO, I want non-bypassable safety controls that prevent catastrophic agent actions, so that rogue or misconfigured agents cannot execute destructive operations in production.

#### Acceptance Criteria

1. THE TealEngine SHALL support a `PLAN_ONLY` runtime mode where all side-effecting actions are denied regardless of policy evaluation outcome.
2. WHILE the TealEngine is in `PLAN_ONLY` mode, THE Policy_Decision_Point SHALL deny any action classified as side-effecting with reason code `PLAN_ONLY_BLOCK`.
3. WHILE the TealEngine is in `PLAN_ONLY` mode, THE Policy_Decision_Point SHALL allow read-only and reasoning actions to proceed normally.
4. THE TealEngine SHALL support `FREEZE` rules that are immutable at runtime and cannot be overridden by agent output, policy hot-swap, or configuration change.
5. WHEN an action matches an active FREEZE rule, THE Policy_Decision_Point SHALL deny the action with reason code `FREEZE_BLOCK` regardless of any other policy evaluation.
6. THE TealEngine SHALL provide a high-impact action pack defining action categories: database writes, infrastructure mutations, production deployments, secrets reveal, and code merges to protected branches.
7. WHEN an action matches a high-impact action category and no explicit approval has been recorded, THE Policy_Decision_Point SHALL deny the action with reason code `HIGH_IMPACT_APPROVAL_REQUIRED`.
8. THE FREEZE rules SHALL persist across policy bundle hot-swaps and process restarts.
9. IF an agent output attempts to modify or disable a FREEZE rule, THEN THE TealEngine SHALL ignore the modification attempt and log a `FREEZE_TAMPER_ATTEMPT` event.

### Requirement 5: SOC/IR Evidence Pipeline and Response Hooks (Epic E)

**User Story:** As a SOC analyst, I want governance decisions exported as structured signals to SIEM and observability systems, so that autonomous agent actions are visible in existing security monitoring workflows.

#### Acceptance Criteria

1. THE TealEngine SHALL export governance decisions in a SIEM-compatible structured log format containing: timestamp, decision outcome, reason codes, policy version, agent identity, action type, and risk score.
2. THE TealEngine SHALL emit OpenTelemetry spans for each governance evaluation with attributes conforming to a documented span convention.
3. WHEN a policy violation occurs, THE TealEngine SHALL invoke configured response hooks within 100 milliseconds of the decision.
4. THE TealEngine SHALL support deduplication of response hook invocations for repeated violations of the same rule within a configurable time window.
5. THE TealEngine SHALL support rate-limiting of response hook invocations to prevent alert storms during sustained attack patterns.
6. THE TealEngine SHALL include a correlation identifier in all exported decision logs that links to the originating agent request trace.
7. IF a response hook invocation fails, THEN THE TealEngine SHALL retry the invocation according to a configurable retry policy and log the failure.

### Requirement 6: Standards-Aligned Policy Packs — OWASP Agentic Top 10 (Epic F)

**User Story:** As a compliance officer, I want prebuilt policy packs mapped to OWASP Agentic Top 10 categories, so that the organization can demonstrate standards-aligned governance with minimal customization.

#### Acceptance Criteria

1. THE TealEngine SHALL ship a policy pack at path `policy-packs/owasp-agentic-top10/` containing controls for each of the 10 OWASP Agentic categories.
2. THE policy pack SHALL assign a stable reason code to each control that maps to the corresponding OWASP category identifier.
3. THE policy pack SHALL include a standards-mapping document that traces each control to its OWASP source requirement.
4. THE policy pack SHALL include evidence templates that produce audit artifacts aligned with the OWASP category structure.
5. WHEN a policy pack control triggers, THE Decision SHALL include the OWASP category identifier in its metadata.
6. THE policy pack SHALL be loadable with zero configuration beyond specifying the pack name in TealEngine initialization.
7. THE policy pack SHALL be extensible, allowing teams to add custom controls without modifying the shipped pack files.

### Requirement 7: TealProof — Cryptographic Governance Receipts (Epic G)

**User Story:** As an auditor, I want cryptographic proof that an AI agent operated within policy, so that governance compliance is mathematically verifiable rather than dependent on editable logs.

#### Acceptance Criteria

1. THE TealProof module SHALL compute a hash of each governance decision combining: decision outcome, request context, timestamp, policy version, and the previous decision hash.
2. THE TealProof module SHALL organize decision hashes into a Merkle tree structure where each leaf represents one governance decision.
3. THE TealProof module SHALL periodically anchor the Merkle tree root to an RFC 3161 timestamping service.
4. THE TealProof module SHALL produce compact verification proofs that allow a third party to verify a specific decision was included in an anchored Merkle root without access to the full tree.
5. THE TealProof module SHALL generate a Governance Passport for each agent containing a compact cryptographic summary of its policy-compliant execution history.
6. THE TealProof module SHALL ship a standalone Verification SDK that verifies governance proofs without requiring a TealTiger installation.
7. IF the RFC 3161 timestamping service is unavailable, THEN THE TealProof module SHALL queue Merkle roots for anchoring and continue producing local proofs.
8. WHEN a decision hash does not match the expected chain (indicating tampering), THE TealProof module SHALL emit a `PROOF_CHAIN_INTEGRITY_VIOLATION` event.

##### 7.2 Verification Levels and Negative-Case Handling

9. THE TealProof Verification SDK SHALL define three explicit verification levels: `LEVEL_1_INTEGRITY` (receipt cryptographically valid), `LEVEL_2_SUFFICIENCY` (receipt + trace + policy confirmed), and `LEVEL_3_APPROPRIATENESS` (receipt + trace + policy + business context verified). The verifier SHALL declare which level it can attest to and SHALL NOT over-claim.
10. WHEN the policy package hash in a receipt does not match the policy bundle presented for verification, THE Verification SDK SHALL return `PROOF_POLICY_MISMATCH` indicating integrity holds but correct policy cannot be confirmed.
11. WHEN the evaluated attributes hash in a receipt does not match the attributes presented for verification, THE Verification SDK SHALL return `PROOF_ATTRIBUTE_INTEGRITY_FAILED`.
12. WHEN a Merkle leaf is omitted from an exported proof set (detected via leaf sequence numbering), THE Verification SDK SHALL return `PROOF_COVERAGE_INCOMPLETE` — the proof set is valid for what it contains but cannot claim completeness.
13. WHEN a receipt timestamp is valid but the gap between decision timestamp and policy bundle timestamp exceeds a configurable freshness threshold, THE Verification SDK SHALL return `POLICY_STALENESS_WARNING`.
14. WHEN the subject or correlation ID in a receipt cannot be resolved against the underlying agent/tool trace, THE Verification SDK SHALL return `INTEGRITY_VERIFIED_CONTEXT_UNRESOLVABLE` rather than claiming full verification.
15. THE Verification SDK SHALL never return a verification result that implies decision appropriateness when only cryptographic integrity can be confirmed. A receipt can be valid and still not prove the decision was appropriate for the operating context.

### Requirement 8: TealFlow — Declarative Governance Workflows (Epic H)

**User Story:** As a governance team lead, I want to express governance policies as composable, event-driven YAML workflows, so that non-developers can read, modify, and audit governance rules without touching application code.

#### Acceptance Criteria

1. THE TealFlow engine SHALL parse workflow YAML documents conforming to a defined schema with fields: `name`, `on` (triggers), `env`, `jobs`, and job-level `steps`, `if`, and `needs`.
2. THE TealFlow engine SHALL support event triggers: `agent_action` (with type filters), `schedule` (cron syntax), `workflow_dispatch` (manual), and `policy_violation`.
3. THE TealFlow engine SHALL execute steps sequentially within a job and execute independent jobs in parallel.
4. THE TealFlow engine SHALL support `needs` dependencies between jobs, executing dependent jobs only after their prerequisites complete successfully.
5. THE TealFlow engine SHALL support `if` conditional expressions that evaluate against action attributes, risk scores, environment variables, and agent identity.
6. THE TealFlow engine SHALL support a `uses` syntax for referencing reusable governance actions from a local or remote registry.
7. THE TealFlow engine SHALL enforce org-level workflow inheritance where team-level workflows extend but cannot weaken org-level controls.
8. WHEN a team-level workflow attempts to remove or weaken an org-level control, THE TealFlow engine SHALL reject the workflow with reason code `WORKFLOW_FLOOR_VIOLATION`.
9. THE TealFlow engine SHALL support encrypted secrets referenced in workflow steps without exposing secret values in logs or decision artifacts.
10. THE TealFlow Pretty_Printer SHALL format TealFlow Workflow objects back into valid YAML documents.
11. FOR ALL valid TealFlow Workflow objects, parsing then printing then parsing SHALL produce an equivalent object (round-trip property).

### Requirement 9: OWASP Top 10 for LLMs 2026 Gap Closure (Epic I)

**User Story:** As a security engineer, I want TealTiger to close identified gaps against the OWASP Top 10 for LLM Applications 2026, so that the SDK provides strong or comprehensive coverage for 9 out of 10 entries.

#### Acceptance Criteria

##### 9.1 Unicode Normalization (Gap 10)

1. WHEN untrusted content enters model context, THE TealGuard module SHALL strip or reject Tag-block characters (U+E0000–U+E007F), variation-selector characters (U+FE00–U+FE0F), and zero-width characters (U+200B, U+200C, U+200D, U+2060).
2. THE TealGuard module SHALL apply Unicode normalization at every boundary where untrusted content is concatenated into model context.

##### 9.2 Encoded Output Detection (Gap 12)

3. WHEN model output contains base64-encoded, hex-encoded, or ROT13-encoded content exceeding a configurable length threshold, THE TealGuard module SHALL flag the output with reason code `ENCODED_OUTPUT_DETECTED`.
4. WHILE the TealEngine is in ENFORCE mode, THE TealGuard module SHALL block outputs flagged with `ENCODED_OUTPUT_DETECTED`.

##### 9.3 Memory Content Sanitization (Gap 7)

5. WHEN a candidate memory entry scores above a configurable instruction-likeness threshold, THE TealMemory module SHALL reject the entry with reason code `MEMORY_INSTRUCTION_INJECTION`.
6. THE TealMemory module SHALL score candidate memory entries for: imperative verb patterns, conditional trigger patterns, role/tool reference patterns, and encoded payload patterns.

##### 9.4 Memory Exfiltration Detection (Gap 8)

7. WHEN stored memory contains URLs with data-bearing query parameters, markdown image links, or webhook-formatted strings pointing to non-allowlisted domains, THE TealMemory module SHALL flag the entry with reason code `MEMORY_EXFILTRATION_RISK`.

##### 9.5 Output Control Character Sanitization (Gap 13)

8. WHEN model output contains ANSI escape sequences, OSC 52 sequences, BEL characters, or non-printable control characters, THE TealGuard module SHALL strip the control characters before the output reaches any interpreting sink.

##### 9.6 Markdown Image Exfiltration Detection (Gap 14)

9. WHEN model output contains markdown image URLs, iframe references, or link-preview triggers pointing to non-allowlisted domains, THE TealGuard module SHALL block the output with reason code `MARKDOWN_EXFILTRATION_DETECTED`.
10. THE TealGuard module SHALL flag outputs containing URLs with data-bearing query parameters (base64-looking values, encoded payloads) regardless of domain allowlist status.

##### 9.7 MCP Definition-Drift Monitoring (Gap 3)

11. THE TealRegistry module SHALL store a cryptographic hash of each approved tool definition (name + description + inputSchema).
12. WHEN a tool invocation occurs and the live tool definition hash does not match the approved hash, THE TealRegistry module SHALL block the invocation with reason code `MCP_DEFINITION_DRIFT`.
13. THE TealRegistry module SHALL scan tool description fields for instruction-injection patterns (Unicode manipulation, imperative verbs, conditional logic).

##### 9.8 Memory Write Provenance Tagging (Gap 6)

14. THE TealMemory module SHALL tag every memory write with a source class using transitive propagation. The candidate's trust level SHALL be the minimum across all sources in its lineage:
    - `direct_user` — explicitly stated by the user in their own words
    - `model_inference` — generated, summarized, or paraphrased by the model. Reflective/self-summarizing outputs are capped at the trust of the trajectory they reflect over.
    - `tool_output_internal` — first-party tool, vetted schema, no third-party passthrough
    - `tool_output_external` — third-party MCP, web-fetching, agent-to-agent. Treated as untrusted_document.
    - `untrusted_document` — external content (web, email, uploaded files, RAG from external corpora, agent observations)
15. THE default policy SHALL be:
    - `direct_user` → allow with sanitization
    - `model_inference` / `tool_output_internal` → sanitize, strip imperatives and conditional triggers; downgraded to lowest-trust ancestor
    - `tool_output_external` / `untrusted_document` → block, or require out-of-band user confirmation with a diff
16. WHEN a memory write candidate's lineage includes any `untrusted_document` or `tool_output_external` source, THE TealMemory module SHALL propagate that trust level transitively regardless of intermediate processing steps.

##### 9.9 Reasoning-Trace Governance (Gap 11)

16. THE TealAudit module SHALL apply PII detection and secret detection to reasoning traces (chain-of-thought, extended-thinking) before the traces reach log sinks.
17. WHERE reasoning-trace redaction is enabled, THE TealAudit module SHALL redact or hash sensitive patterns in reasoning traces before export.

##### 9.10 Multimodal Input Governance (Gap 1)

18. WHEN a non-text input (image, audio, video) is submitted for governance evaluation, THE TealGuard module SHALL extract text content via OCR or transcription and apply existing text-based governance filters to the extracted content.
19. THE TealGuard module SHALL strip metadata from non-text inputs before governance evaluation.

##### 9.11 Behavioral Drift Detection (Gap 4)

20. THE TealDrift module SHALL establish a statistical baseline of agent response patterns (refusal rates, response length distributions, topic distributions) per model and provider.
21. WHEN ongoing agent behavior diverges from the established baseline beyond a configurable threshold, THE TealDrift module SHALL emit a `BEHAVIORAL_DRIFT_DETECTED` alert.

##### 9.12 Compositional Adapter Safety (Gap 2)

22. THE TealRegistry module SHALL support an approved adapter-combination allowlist in addition to individual adapter allowlists.
23. WHEN an adapter composition is loaded that is not in the approved combinations allowlist, THE TealRegistry module SHALL reject the composition with reason code `UNAPPROVED_ADAPTER_COMPOSITION`.

### Requirement 10: TealClassifier — Lightweight ML Detection Layer (Epic J)

**User Story:** As a security engineer, I want an optional local ML classifier that improves prompt injection recall from 50% to 80%+ without adding cloud dependencies or violating the no-LLM-in-governance-path principle, so that the SDK catches attacks that regex patterns miss.

#### Acceptance Criteria

1. THE TealClassifier module SHALL load an ONNX model and perform local inference without external API calls.
2. THE TealClassifier module SHALL support four ensemble modes: `regex_only`, `ml_only`, `ensemble_union`, and `ensemble_intersection`.
3. WHEN ensemble mode is `regex_only`, THE TealGuard module SHALL use only the regex detection layer (v1.2 backward-compatible behavior).
4. WHEN ensemble mode is `ml_only`, THE TealGuard module SHALL use only the ML classifier output for detection decisions.
5. WHEN ensemble mode is `ensemble_union`, THE TealGuard module SHALL block content if either the regex layer or the ML classifier detects a threat.
6. WHEN ensemble mode is `ensemble_intersection`, THE TealGuard module SHALL block content only if both the regex layer and the ML classifier detect a threat.
7. THE TealClassifier module SHALL produce a detection confidence score between 0.0 and 1.0 for each evaluation.
8. THE TealClassifier module SHALL complete inference within 20 milliseconds on CPU for inputs up to 512 tokens.
9. IF the ONNX model fails to load or inference fails, THEN THE TealClassifier module SHALL fall back to regex-only detection and emit a `CLASSIFIER_FALLBACK` event.
10. THE TealClassifier module SHALL support model updates (downloading new ONNX artifacts) without requiring an SDK version upgrade.
11. THE TealClassifier module SHALL produce deterministic outputs for identical inputs (no sampling, no temperature, no stochastic behavior).
12. THE TealClassifier module SHALL achieve 80% or higher recall on the PINT benchmark in `ensemble_union` mode.
13. THE TealClassifier module SHALL maintain 85% or higher precision on the PINT benchmark in `ensemble_union` mode.

### Requirement 11: Backward Compatibility and Contract Stability

**User Story:** As an existing TealTiger v1.2 user, I want to upgrade to v1.3 without breaking changes to my existing integration, so that adoption of new capabilities is incremental and risk-free.

#### Acceptance Criteria

1. THE TealEngine v1.3 SHALL accept all valid v1.2 policy configurations without modification.
2. THE TealEngine v1.3 SHALL produce identical Decision outputs for v1.2 policy configurations as TealEngine v1.2 produced for the same inputs.
3. WHEN no v1.3-specific features are configured, THE TealEngine v1.3 SHALL behave identically to TealEngine v1.2.
4. THE TealClassifier module SHALL default to `regex_only` ensemble mode, preserving v1.2 detection behavior unless explicitly opted in.
5. THE TealFlow engine SHALL be opt-in; existing v1.2 configurations SHALL continue to function without TealFlow workflow files present.
6. THE Decision contract (typed Decision object with action, risk score, reason codes, evidence fields, and policy version) SHALL NOT change its field names, field types, or field semantics from v1.2 to v1.3.
7. THE public API surface (function signatures, method names, parameter types, return types) of all v1.2 modules SHALL remain unchanged in v1.3. New parameters SHALL be optional with defaults that preserve v1.2 behavior.
8. THE configuration schema for all v1.2 modules SHALL remain backward compatible. New configuration fields SHALL be optional with defaults that preserve v1.2 behavior.
9. THE TealGuard, TealSecrets, TealAudit, TealCircuit, TealMemory, and TealRegistry modules SHALL accept the same constructor parameters in v1.3 as they accepted in v1.2.
10. IF a v1.3 feature requires a new configuration field, THEN the field SHALL be optional and its absence SHALL produce v1.2-equivalent behavior.
11. THE npm package and PyPI package SHALL follow semantic versioning: v1.3.0 indicates new features with no breaking changes to the public API contract.

### Requirement 14: Agentic Platform Integrations

**User Story:** As an enterprise architect deploying AI agents on cloud platforms, I want TealTiger to integrate natively as governance middleware within AWS Bedrock Agents, AWS AgentCore, and Azure AI Agent Service, so that governance is enforced within the agent runtime — not just at the API call layer.

#### Acceptance Criteria

##### 14.1 AWS Bedrock Agents Integration

1. THE TealEngine SHALL provide a Bedrock Agents guardrail adapter that conforms to the Bedrock Guardrails API contract.
2. THE Bedrock Agents adapter SHALL evaluate agent actions (tool invocations, knowledge base queries, response generation) through TealTiger's governance pipeline before execution.
3. THE Bedrock Agents adapter SHALL return ALLOW or DENY decisions in the format expected by the Bedrock Agents runtime.
4. THE Bedrock Agents adapter SHALL support deployment as a Lambda-backed action group or as a custom guardrail.

##### 14.2 AWS AgentCore Integration

5. THE TealEngine SHALL provide an AgentCore governance plugin that hooks into the agent lifecycle at pre-action and post-action stages.
6. THE AgentCore plugin SHALL evaluate tool calls, memory writes, and inter-agent messages through TealTiger's governance pipeline.
7. THE AgentCore plugin SHALL support the AgentCore plugin interface without requiring modifications to the agent's core logic.
8. THE AgentCore plugin SHALL propagate TealTiger's correlation IDs into AgentCore's observability traces.

##### 14.3 Azure AI Agent Service Integration

9. THE TealEngine SHALL provide an Azure AI Agent Service middleware that integrates into the agent's tool-call pipeline.
10. THE Azure middleware SHALL evaluate tool invocations and content generation through TealTiger's governance pipeline before execution.
11. THE Azure middleware SHALL support deployment as an Azure Functions-based middleware or as an in-process SDK integration.
12. THE Azure middleware SHALL integrate with Azure Monitor and Application Insights for governance decision telemetry.

##### 14.4 Cross-Platform Requirements

13. ALL agentic platform integrations SHALL use the same TealEngine policy evaluation logic — no platform-specific policy behavior.
14. ALL agentic platform integrations SHALL produce identical Decision objects for identical inputs regardless of which platform the agent runs on.
15. ALL agentic platform integrations SHALL support the full TealTiger module set (TealGuard, TealSecrets, TealAudit, TealCircuit, TealMemory, TealRegistry, TealProof).
16. THE agentic platform integrations SHALL NOT require changes to the core TealEngine — they are adapters that translate between platform-specific interfaces and TealTiger's governance API.

### Requirement 15: Governance Tamper Resistance and Separation of Duties

**User Story:** As a CISO, I want the SDK to enforce separation of duties between governance teams (who author policies) and developers (who integrate the SDK), so that application code cannot weaken, bypass, or tamper with governance enforcement.

#### Acceptance Criteria

##### 15.1 Anti-Tamper Controls

1. THE SDK SHALL NOT expose any configuration flag, environment variable, or API parameter that disables, weakens, or bypasses governance enforcement.
2. WHEN application code attempts to set an unknown or forbidden configuration key (e.g., `disable_enforcement`, `bypass`, `allow_all`, `permissive_mode`), THE SDK SHALL log a `TAMPER_ATTEMPT` security event and ignore the setting.
3. THE SDK SHALL NOT load governance policies from local file paths in production mode. Policies SHALL only be sourced from an approved registry.
4. THE SDK SHALL validate the integrity (hash or signature) of every governance bundle on load. IF validation fails, THE SDK SHALL emit a `BUNDLE_INTEGRITY_FAILURE` event and follow the governance-defined fail behavior.
5. THE SDK SHALL NOT allow application code to override, modify, or selectively skip individual policy decisions after evaluation.
6. WHEN the SDK returns a DENY decision, THE SDK SHALL NOT provide any mechanism for application code to convert that decision to ALLOW without a governance-approved exception.

##### 15.2 Separation of Duties

7. THE governance artifact authoring process SHALL be separate from the application development process. Governance teams own write/approve rights; application teams have read-only access to artifacts.
8. THE SDK SHALL bind workload identity (app_id, environment, tenant_id) to every governance evaluation. Identity SHALL be derived from platform-level identity (K8s ServiceAccount, IAM role, workload certificate), not freely set by developers at runtime.
9. THE SDK SHALL include the governance bundle version, policy ID, and artifact hash in every TEEC envelope, enabling auditors to verify exactly which governance was in effect at any point in time.
10. THE fail behavior (fail-open vs fail-closed) SHALL be defined in governance artifacts, not in application configuration. Developers SHALL NOT control whether the system fails open or closed.

##### 15.3 Registry Security

11. THE SDK SHALL only connect to registry endpoints that are present in a platform-managed allow-list.
12. WHEN a registry endpoint is not in the allow-list, THE SDK SHALL refuse to load artifacts and emit a `REGISTRY_ENDPOINT_VIOLATION` event.
13. THE SDK SHALL support pinned bundle versions only when the pin references an approved version and pin changes require platform-level change control.

##### 15.4 Capability Negotiation and Downgrade Prevention

14. THE SDK SHALL export a capabilities manifest declaring which governance features it supports.
15. WHEN a governance bundle declares required capabilities that the SDK version does not support, THE SDK SHALL refuse to load the bundle and emit a `CAPABILITY_MISMATCH` event.
16. THE SDK SHALL NOT allow application code to force loading of bundles that require unsupported capabilities.

### Requirement 16: Enterprise Governance Operating Model and Process Documentation

**User Story:** As a governance program manager, I want TealTiger to provide clear documentation of personas, workflows, and change processes for enterprise governance operations, so that organizations can implement TealTiger not just as a technical product but as a complete governance operating model.

#### Acceptance Criteria

##### 16.1 Persona Definitions

1. THE documentation SHALL define the following governance personas with their responsibilities, permissions, and boundaries:
   - **Governance Author** — Creates and maintains governance policies, controls, and evidence requirements. Owns the "what" and "why" of governance.
   - **Governance Approver** — Reviews and approves policy changes before they become enforceable. Ensures compliance with regulatory requirements.
   - **Platform Engineer** — Manages the registry, SDK deployment, identity binding, and infrastructure. Owns the "where" and "how" of enforcement.
   - **Application Developer** — Integrates the SDK into agent applications. Consumes governance decisions. Cannot modify governance intent.
   - **Auditor / Compliance Reviewer** — Verifies governance was enforced correctly. Reviews TEEC evidence, decision trails, and policy history.
   - **Incident Responder** — Investigates governance violations and evidence gaps. Uses audit trails to reconstruct what happened.

2. THE documentation SHALL include a RACI matrix mapping each governance activity to the responsible persona.

##### 16.2 Policy Lifecycle and Change Process

3. THE documentation SHALL define the complete policy lifecycle with states: `DRAFT` → `REVIEW` → `APPROVED` → `ENFORCED` → `DEPRECATED` → `RETIRED`.
4. THE documentation SHALL define the change process for creating a new policy:
   - Governance Author drafts policy with intent, scope, controls, and evidence requirements
   - Governance Approver reviews and approves (minimum 2-person approval for production policies)
   - Platform Engineer publishes approved policy to registry with version tag
   - SDK pulls new bundle on next refresh cycle
   - Policy becomes enforceable per its configured rollout mode (REPORT_ONLY → MONITOR → ENFORCE)
5. THE documentation SHALL define the change process for modifying an existing policy:
   - Change request with justification
   - Impact assessment (which agents/apps affected)
   - Approval workflow (same as new policy)
   - Staged rollout with rollback capability
6. THE documentation SHALL define the exception process:
   - Who can request an exception
   - Who approves (must be different from requester)
   - Mandatory expiry date
   - Audit trail of exception lifecycle
   - Automatic re-evaluation at expiry

##### 16.3 Operational Workflows

7. THE documentation SHALL include workflow diagrams for:
   - Day-0: Initial TealTiger deployment and first policy bundle
   - Day-1: Adding a new governance policy to an existing deployment
   - Day-2: Responding to a governance violation alert
   - Emergency: Activating a FREEZE rule in response to an incident
   - Rollback: Reverting to a previous policy bundle version

8. THE documentation SHALL include a "Quick Start for Governance Teams" guide that enables a governance team to author and deploy their first policy without writing code.

##### 16.4 Audit and Evidence Trail

9. THE documentation SHALL describe how auditors can:
   - Verify which policy version was active at a specific timestamp
   - Trace a governance decision back to the policy, control, and evidence that produced it
   - Detect gaps in evidence (missing TEEC envelopes indicating potential bypass)
   - Reconstruct the governance state during an incident window

10. THE documentation SHALL include example audit scenarios with sample TEEC envelopes and evidence artifacts.

##### 16.5 Enterprise Adoption Guide

11. THE documentation SHALL include a phased adoption guide:
    - Phase 1: REPORT_ONLY mode (observe without blocking — build confidence)
    - Phase 2: MONITOR mode (alert on violations — validate policies)
    - Phase 3: ENFORCE mode (block violations — full governance)
12. THE documentation SHALL address common enterprise objections (FAQ format):
    - "What if it blocks legitimate traffic?"
    - "How do we handle false positives?"
    - "What happens during an outage?"
    - "How do we prove compliance to regulators?"
    - "What's the performance impact?"

##### 16.6 Enterprise Adoption Models

13. THE documentation SHALL define two supported adoption models:
    - **Model 1: Governance-First (recommended)** — Platform/governance team onboards TealTiger, defines baseline policies centrally, distributes a pre-governed SDK to developers. Developers consume governance as read-only.
    - **Model 2: Developer-First (transitional)** — Developers integrate the SDK directly with default policies, then governance is introduced centrally and the SDK is progressively locked down.

### Requirement 17: Enhanced Cost Governance (FinOps-Aligned)

**User Story:** As a FinOps lead, I want AI cost limits to be governance-owned artifacts (not developer-configured parameters), with per-agent attribution, anomaly detection, and auditable cost decisions, so that AI spend is governed with the same rigor as other enterprise cost controls.

#### Acceptance Criteria

##### 17.1 Cost Limits as Governance Artifacts

1. THE cost budget configuration (per-request, per-session, per-daily, per-agent limits) SHALL be definable in governance policy bundles pulled from the registry.
2. WHEN cost limits are defined in a governance bundle, THE SDK SHALL enforce those limits regardless of any cost configuration in application code.
3. THE governance-defined cost limits SHALL take precedence over application-defined cost limits (governance floor enforcement).
4. THE SDK SHALL NOT allow application code to raise cost limits above governance-defined ceilings.

##### 17.2 Per-Agent and Per-Workflow Cost Attribution

5. THE cost tracking module SHALL support cost attribution at the agent level (total cost per agent identity over configurable time windows).
6. THE cost tracking module SHALL support cost attribution at the workflow level (total cost per workflow/session across multiple requests).
7. THE cost tracking module SHALL emit cost attribution data in a format consumable by external FinOps platforms (structured JSON with agent_id, workflow_id, provider, model, token counts, cost).

##### 17.3 Cost Anomaly Detection

8. THE cost tracking module SHALL maintain a rolling baseline of cost-per-request for each agent and provider combination.
9. WHEN a single request's cost exceeds a configurable multiple of the rolling baseline (e.g., 10x), THE SDK SHALL emit a `COST_ANOMALY_DETECTED` alert before executing the request.
10. WHEN an agent's cumulative session cost growth rate exceeds a configurable threshold, THE SDK SHALL emit a `COST_SPIKE_DETECTED` alert.

##### 17.4 Reasoning-Token Cost Governance

11. THE cost tracking module SHALL separately track and report thinking/reasoning tokens (extended-thinking, chain-of-thought) distinct from completion tokens.
12. THE governance bundle SHALL support separate budget limits for reasoning tokens (e.g., `max_thinking_tokens_per_request`, `max_thinking_cost_per_session`).
13. WHEN reasoning-token limits are exceeded, THE SDK SHALL deny the request with reason code `REASONING_TOKEN_BUDGET_EXCEEDED`.

##### 17.5 Cost Evidence in TEEC Envelopes

14. EVERY cost-related enforcement decision (allow within budget, deny over budget, anomaly alert) SHALL produce a TEEC-compatible evidence record containing: policy version, budget limit, actual cost, remaining budget, and decision outcome.
15. THE cost evidence SHALL be queryable by auditors to answer: "Why was this request allowed/denied from a cost perspective, and which policy version governed it?"

### Requirement 18: State and Temporal Governance Modules

**User Story:** As a platform engineer, I want governance controls over agent context size, session duration, and temporal boundaries, so that agents cannot accumulate unbounded state or operate beyond approved time windows.

#### Acceptance Criteria

##### 18.1 TealState — Context and State Governance

1. THE TealState module SHALL enforce configurable maximum context window size limits per agent and provider.
2. WHEN an agent's accumulated context exceeds the configured size limit, THE TealState module SHALL emit a `CONTEXT_SIZE_EXCEEDED` event and take the governance-defined action (truncate, deny, or alert).
3. THE TealState module SHALL track provenance metadata for context entries (source, timestamp, trust tier).

7. THE TealTemporal module SHALL support cooldown periods between high-impact actions (configurable minimum interval between consecutive actions of the same type).
8. THE TealTemporal module SHALL support time-of-day restrictions (e.g., block production deployments outside business hours) configurable via governance artifacts.
9. THE TealTemporal module SHALL support session age tracking and emit `SESSION_AGE_WARNING` events at configurable thresholds before TTL expiry.

### Requirement 19: Agent Identity Integrity and Zero Standing Privilege

**User Story:** As a security architect, I want to verify the integrity of AI agents making requests via signed attestations and enforce that no agent has permanent standing access to tools or resources, so that impersonation attacks are prevented and the blast radius of compromised agents is minimized.

#### Acceptance Criteria

##### 19.1 Agent Identity Integrity (AUTH.AGENT.INTEGRITY)

1. THE TealEngine SHALL support agent identity verification via signed attestations or hardware-bound tokens before allowing execution.
2. WHEN an agent request lacks a valid attestation and the policy requires attestation, THE TealEngine SHALL deny the request with reason code `AGENT_ATTESTATION_MISSING`.
3. WHEN an agent attestation signature is invalid or expired, THE TealEngine SHALL deny the request with reason code `AGENT_INTEGRITY_FAILED`.
4. THE TealEngine SHALL maintain a list of trusted attestation signers configurable via governance artifacts.
5. THE agent integrity verification SHALL be independent of the agent's self-reported identity — attestation must come from an external trust anchor (platform identity, hardware token, or signed certificate).

##### 19.2 Zero Standing Privilege (AUTH.ACCESS.ZSP)

6. THE TealEngine SHALL support a Zero Standing Privilege mode where no agent has permanent access to tools, models, or resources.
7. WHEN Zero Standing Privilege mode is enabled, ALL agent access to tools and resources SHALL be Just-In-Time (JIT) granted per session or per request with an explicit expiry.
8. WHEN a JIT access grant expires, THE TealEngine SHALL deny subsequent requests from that agent with reason code `ACCESS_GRANT_EXPIRED`.
9. WHEN an agent attempts to use a tool or resource without a valid JIT grant and ZSP mode is enabled, THE TealEngine SHALL deny the request with reason code `ACCESS_STANDING_PRIVILEGE_DENIED`.
10. THE maximum JIT grant TTL SHALL be configurable via governance artifacts (e.g., `authority.access.max_grant_ttl_ms`).
11. THE TealEngine SHALL log all JIT grant issuances and expirations in the audit trail with the granting identity, scope, and TTL.

### Requirement 20: Portability and Migration — Vendor Lock-In Reduction

**User Story:** As an enterprise platform architect, I want TealTiger to minimize vendor lock-in by providing portable policy formats, standard interfaces, and clear migration paths, so that my organization can adopt TealTiger with confidence that switching costs remain manageable.

#### Acceptance Criteria

##### 20.1 Portable Policy Export

1. THE TealEngine SHALL support exporting governance policies to OPA-compatible Rego format, enabling organizations to migrate policy logic to Open Policy Agent without rewriting from scratch.
2. THE policy export SHALL preserve policy intent, scope, and decision semantics in the target format.
3. THE policy export SHALL include a mapping document that traces each exported rule back to its TealTiger source policy and control ID.
4. THE policy export SHALL support batch export of all active policies in a governance bundle.

##### 20.2 Standard Governance Interface

5. THE TealEngine SHALL expose a `GovernanceProvider` interface that abstracts the evaluation engine, allowing organizations to implement alternative backends behind the same API surface.
6. THE `GovernanceProvider` interface SHALL define: `evaluate(context) → Decision`, `loadPolicies(bundle) → void`, and `getCapabilities() → CapabilityManifest`.
7. THE SDK SHALL document the `GovernanceProvider` interface as a public, stable contract with semantic versioning guarantees.

##### 20.3 Open Context Schema

8. THE TealEngine SHALL publish its evaluation context schema (the input shape passed to `evaluate()`) as an open, versioned JSON Schema document.
9. THE published context schema SHALL be usable by third-party governance tools to accept TealTiger-formatted evaluation requests without modification.
10. THE context schema SHALL be versioned independently of the SDK version, with backward-compatible evolution guarantees.

##### 20.4 Module Standalone Usability

11. THE TealGuard, TealSecrets, TealMemory, TealRegistry, TealCircuit, TealAudit, TealProof, TealClassifier, TealDrift, TealState, and TealTemporal modules SHALL each be importable and usable as standalone packages without requiring the full TealEngine.
12. EACH standalone module SHALL expose a documented public API that does not depend on TealEngine internals.
13. THE standalone module APIs SHALL be stable across minor versions (no breaking changes within v1.x).

##### 20.5 Migration Documentation

14. THE documentation SHALL include a "Migration Guide" describing how to migrate from TealTiger to: (a) raw OPA/Rego, (b) Cedar, (c) custom policy engines.
15. THE migration guide SHALL identify which TealTiger capabilities are portable (policy logic, audit data, evidence artifacts) and which require reimplementation (domain-specific modules like TealGuard, TealSecrets).
16. THE migration guide SHALL include estimated effort levels (low/medium/high) for each migration path.
17. THE documentation SHALL include a "Portability FAQ" addressing common enterprise lock-in concerns.

##### 20.6 Data Portability

18. ALL audit logs, TEEC evidence envelopes, and governance receipts SHALL be exportable in open, documented formats (JSON, SARIF, JUnit XML) that do not require TealTiger tooling to parse.
19. THE TealProof verification SDK SHALL remain usable independently of the TealTiger SDK, enabling proof verification after migration away from TealTiger.
20. THE cost attribution data SHALL be exportable in a format compatible with standard FinOps platforms without TealTiger-specific tooling.

### Requirement 21: Ecosystem Defensibility and Standards Positioning

**User Story:** As the TealTiger product owner, I want the SDK to establish ecosystem-level defensibility through framework neutrality, governance telemetry ownership, and standards influence, so that TealTiger becomes the default governance layer that is difficult to replace — not because of lock-in, but because of ecosystem integration depth.

#### Acceptance Criteria

##### 21.1 Framework Neutrality as First-Class Requirement

1. THE TealEngine SHALL provide documented, tested integration adapters for: LangChain, LangGraph, CrewAI, MCP (Model Context Protocol), OpenAI SDK (direct), Anthropic SDK (direct), and custom agent runtimes.
2. EACH integration adapter SHALL require no more than 5 lines of code to add governance to an existing agent application.
3. THE integration adapters SHALL NOT require the host application to inherit from TealTiger base classes or use TealTiger-specific orchestration primitives.
4. THE SDK SHALL publish an "Integration Adapter Specification" enabling third parties to build adapters for new frameworks without TealTiger team involvement.

##### 21.2 Governance Telemetry as Ecosystem Standard

5. THE TealEngine SHALL define and publish an open "Governance Event Schema" (versioned JSON Schema) for all governance decision events, violations, and audit signals.
6. THE Governance Event Schema SHALL be designed for adoption by other governance tools — not proprietary to TealTiger.
7. THE TealEngine SHALL emit governance telemetry conforming to OpenTelemetry semantic conventions, enabling native integration with any OTel-compatible observability platform.
8. THE SDK SHALL publish governance span conventions as a proposed OTel semantic convention extension, positioning TealTiger's telemetry format as an industry reference.

##### 21.3 Open Governance Standards Contribution

9. THE TealTiger project SHALL publish its policy schema, evidence format (TEEC), and governance event schema as open specifications under Apache 2.0 license, independent of the SDK implementation.
10. THE open specifications SHALL be versioned and maintained in a dedicated repository separate from the SDK, signaling that they are community standards — not vendor artifacts.
11. THE TealFlow workflow schema SHALL be published as an open specification that other governance tools can implement, creating ecosystem lock-in to the format rather than the implementation.

##### 21.4 Enterprise Operational Tooling (Moat Layer)

13. THE TealDashboard SHALL support RBAC aligned with the governance personas defined in Requirement 16.

15. THE TealDashboard SHALL be deployable as a self-hosted solution (no SaaS dependency), preserving the open-source trust model.

##### 21.5 Competitive Positioning Against Proxy-Only Solutions

16. THE documentation SHALL include a "Why Runtime Governance" technical brief explaining the semantic context gap that proxy/gateway solutions cannot bridge (agent state, memory access, tool-chain intent, execution graphs, authorization semantics).
17. THE documentation SHALL include benchmark comparisons showing governance decisions that require runtime context unavailable to network proxies.
18. THE SDK SHALL support a hybrid deployment model where TealTiger handles Layer 2 runtime governance while integrating with Layer 1 infrastructure governance (API gateways, service meshes) for rate limiting, egress control, and DLP.

### Design Constraints (for design phase)
